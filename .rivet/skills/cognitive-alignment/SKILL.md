# 认知对齐前置闸门（Cognitive Alignment Gate）

核心原则：**跑错方向比不跑更浪费时间。**

本 Skill 不生成方案、不写代码、不做设计。它只做一件事：确保 AI 和用户在同一个问题模型里，再进入执行。

## 架构映射

本 Skill 对应 OpenMythos Recurrent-Depth Transformer 的 Prelude 层：编码输入、保存意图、标注层级，为后续 Recurrent Loop（deep-brainstorm 三轮循环）提供稳定的输入锚点。Mythos 的核心机制是"input injection at every loop step prevents drift"——本 Skill 产出的意图原文和问题层级就是那个被反复注入的编码输入 e。

---

## 何时使用

- 复杂功能请求（涉及多个子系统或链路）
- 阶段性规划（Phase 级别的目标讨论）
- 用户说"基于 X/Y/Z"但 X/Y/Z 之间有隐含演进链
- 用户描述的问题可能在多个层级修复
- 任何你不确定"我们到底在解决哪一层问题"的时候
- deep-brainstorm 或 brainstorming 开始前的前置步骤

## 不适用

- 明确的 bug 修复（错误信息清晰、修复位置确定）
- 单文件编辑
- 用户明确说"直接做"

## 执行协议

### Stage 0.1：意图原文保存

**记录用户原话，不改写成工程任务。**

输出格式：
~~~
用户原始意图：
「<逐字复述用户的原话，包括模糊部分和情绪词>」
~~~

禁止：
- 把"优化前置资产利用率"改写成"修 openingWritingBrief handoff"
- 把"让方向更多样"改写成"加 hookType 去重"
- 把模糊意图提前收敛成具体工程点

### Stage 0.2：问题层级标注

**判断当前问题涉及以下哪些层级，标注主层级和关联层级。**

| 层级 | 说明 | 示例 |
|------|------|------|
| L1 业务目标 | 用户想达成什么业务结果 | P24 是前置资产利用率验证 |
| L2 链路结构 | 哪些模块/步骤之间的接线有问题 | 世界/角色资产是否进入 P21/P22 步骤 |
| L3 资产投影 | 资产是否变成可写素材 | assetProjection 是否包含世界制度碎片 |
| L4 选择评审 | 评审/选择机制是否奖励健康方向 | S4 是否奖励非模板方向 |
| L5 执行 handoff | 步骤间数据传递是否完整 | openingWritingBrief 是否传到 Chapter |
| L6 prompt 安全 | prompt 边界是否安全 | XML 标签注入风险 |
| L7 代码实现 | 字段、测试、构建 | 类型定义、Jest、tsc |

输出格式：
~~~
问题层级：
  主层级：L2 链路结构 — <一句话说明为什么>
  关联层级：L3 资产投影、L5 执行 handoff
  不应在此轮处理：L7 代码实现（等链路方向确定后再落代码）
~~~

**反局部补丁检查：** 如果你的方案是以下形式，必须暂停回到本阶段：
- 加一组项目级禁词
- 给某个当前样本硬锁细节
- 对所有内容 blanket escape
- 只修最近的字段透传
- 用当前 accepted 结果替代链路验证

### Stage 0.3：证据时间线标注

**标注当前可用证据属于哪个时间点，能证明什么、不能证明什么。**

| 证据类型 | 含义 |
|----------|------|
| baseline | 改造前的旧链路输出 |
| post-change validation | 改造后的新链路输出 |
| fixture | 合成的测试数据 |
| live | 真实运行的产出 |
| preflight | 预检但未实际执行 |

输出格式：
~~~
证据时间线：
  当前证据：<runId/文件/commit> — 类型：baseline
  能证明：旧链路在无世界资产接入时的方向质量
  不能证明：世界/角色资产接线后的创意质量提升
  下一步需要：post-change validation（接线后重测）
~~~

### Stage 0.4：关键词扩展搜索

**对用户输入中的关键阶段、模块、概念做相邻扩展，避免被首句关键词锚定。**

执行步骤：
1. 提取用户输入中的显性关键词（阶段号、模块名、概念名）
2. 对每个关键词查：
   - 前后相邻 phase/commit（不只查显性关键词本身）
   - 相关 commit 历史（`git log --oneline --grep="<keyword>"` 或 `.planning/` 目录）
   - 中间补丁和过渡计划（phase 之间的未命名改动）
   - 用户没明说但从历史演进可推出的下一条链路
3. 输出扩展发现
4. 如果任务涉及创作方向生成，必须查 `packages/prompts/src/seed-packs/` 目录确认可用方向空间（当前 13 个 seed pack），避免在已有方向空间之外凭空想象

输出格式：
~~~
关键词扩展：
  显性关键词：P21, P22, P23
  扩展发现：
    - P22→P23 之间已有世界引擎基于 P21/P20 的链路接入（commit xxx）
    - P23 之后 P25 补了叙事技巧矩阵（8 种切入技法）
    - 叙事引擎（Phase 16/17）尚未接入 Opening/Chapter
  推导的下一条链路：叙事引擎、节奏骨架是否也应接入？
~~~

### Stage 0.5：认知同步摘要

**用 3-5 句话向用户同步你的理解，等用户确认后再进入执行。**

输出格式：
~~~
认知同步：
  我理解现在不是要 <局部补丁描述>，而是要 <用户真实目标>。
  当前证据属于 <证据类型>，不能用于证明 <什么>。
  我会先 <正确的第一步>，不会先 <应避免的局部补丁>。
  
  确认后我将进入 <下一个 Skill/执行步骤>。
~~~

**等用户确认后才能进入方案、设计或代码。**

### Stage 0.6：创作路径推荐（仅创作方向生成任务适用）

如果当前任务涉及创作方向/题材脑洞生成，基于用户输入特征推荐创作路径：

| 用户输入特征 | 推荐路径 | 说明 |
|-------------|---------|------|
| 包含世界级概念（"如果XX世界"、"一个XX的世界观"） | 路径 A：世界先行 | Sanderson/刘慈欣模式，因果解耦 |
| 包含角色概念（"一个XX的主角"、"XX身份的人"） | 路径 B：角色先行 | 富坚/Gilligan 模式，角色驱动 |
| 模糊或包含多个不相关碎片 | 路径 C：情境先行 | King/Gaiman 模式，what-if 种子涌现 |
| 同时包含世界和角色 | 询问用户 | "哪个对你更重要——这个世界还是这个角色？" |

输出格式：
~~~
创作路径推荐：
  用户输入特征：<分析>
  推荐路径：<A/B/C>
  理由：<一句话>
  
  确认后进入对应路径的 skill（causal-decoupling-engine 路径 A / deep-brainstorm 路径 B/C）。
~~~

如果当前任务不涉及创作方向生成，跳过此步。

## 偏差自检清单

完成 Stage 0.1-0.5 后，逐条检查：

1. 我是否把用户的长期目标压缩成了当前最容易执行的任务？
2. 我是否把链路缺陷处理成了项目级禁词？
3. 我是否把接线前 baseline 当成了接线后验证？
4. 我是否用安全最大化破坏了语义结构？
5. 我是否在没有画资产流向图的情况下开始改代码？
6. 我是否只修了最近的症状，而没有定位责任链路？
7. 我是否被用户首句里的关键词锚定，只查了显性阶段？
8. 我是否把本来能做到 200 的创造性链路优化，压缩成了中规中矩的执行？

如果任何一条为"是"，回到对应 Stage 重新执行。

## 与其他 Skill 的关系

- **brainstorming**：复杂任务时，brainstorming 开始前应先调用 cognitive-alignment
- **deep-brainstorm**：deep-brainstorm 的 Step 0 可以引用 cognitive-alignment 的输出
- **preflight-gate**：preflight-gate 验证执行前置条件，cognitive-alignment 验证认知前置条件；两者互补
- **writing-plans**：cognitive-alignment 确定"解决什么"，writing-plans 确定"怎么做"
