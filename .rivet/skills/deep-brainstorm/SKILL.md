# 深度头脑风暴（Deep Brainstorm）

认知架构：**循环深度推理**（每轮重新注入原始目标防止漂移）+ **达尔文演化算法**（变异 → 选择 → 留存）。

核心原理：好的决策是演化出来的——不是从第一个想到的方案开始细化，而是让多个方案竞争后留下最强的那个。

**强制规则：每次调用必须完成三轮完整思考。不可跳过、合并或缩短。**

---

## TL;DR 速查

```
用户说「深度头脑风暴」「多方案对比」「业务分析」→ 启动 deep-brainstorm
用户说「头脑风暴」「设计方案」→ 用 brainstorming（更轻量）

deep-brainstorm 流程：
  Step 0: 澄清需求 → 启动 2-3 个并行子代理调研
  第一轮（变异）: 生成 4+ 方案，每个占据不同生态位
  第二轮（选择）: 重注入原始目标 → 6 项压力测试 → 灭绝弱方案
  第三轮（适应）: 具体化最强方案 → 实施路径 → 设计文档
  → 用户审查 → 调用 writing-plans
```

**快速判断树：**
- 是「怎么做」（实现层面）？ → 用 `brainstorming`
- 是「做什么/选哪个」（决策层面）？ → 用 `deep-brainstorm`
- 不确定？ → 先问一个澄清问题再决定

## 反高概念铁律（全轮次适用）

高概念是思考的寄生虫——它寄生在抽象层级上，伪装成深度，实际上逃避了具体化。

**判定标准：一个想法是否是高概念？**
- 如果你不能用"谁，在什么场景下，做什么具体动作"来重述它，它就是高概念
- 如果它需要超过一句话来解释其"核心思想"，它就是高概念
- 如果它读起来像商业计划书摘要、咨询报告标题、或公众号爆款文，而不是可执行的方案，它就是高概念

**高概念危险信号：**

| 危险信号 | 示例 | 替换方向 |
|----------|------|----------|
| 抽象战略代替具体动作 | "我们要打造生态闭环" | "第一步：让用户 A 能直接在我们的产品里完成 B 操作，不用跳转到 C" |
| 行业标签代替业务设计 | "这是一个 AI+SaaS 的项目" | "用户打开产品后，输入 X，系统自动做 Y，输出 Z，比原来省了 N 小时" |
| 概念堆砌代替方案对比 | "用区块链解决信任问题" | "具体来说：谁不信任谁？在哪个环节？用区块链替代了哪个现有流程？成本多少？" |
| 愿景代替路径 | "三年内成为行业第一" | "第一个季度：拿到 100 个付费用户。怎么拿？从哪来？" |
| 术语代替洞察 | "利用飞轮效应实现增长" | "用户做完 A 后自然会做 B，B 的结果会吸引新用户来做 A。具体机制是什么？" |

**三轮执行规则：**
- 第一轮（变异）：每个方案必须能用"谁 + 在什么场景 + 做什么 + 得到什么"表达。不能的方案不生成。
- 第二轮（选择）：高概念方案直接灭绝。
- 第三轮（适应）：最终方案的每一步都必须是可执行的动作。不能执行的部分删除重写。

## 0. 触发条件与前置准备

### 触发条件
当收到以下类型的任务时启动：
- 产品方向选择、功能优先级排序
- 技术架构决策、技术选型
- 商业模式设计、定价策略
- 竞品分析、市场定位
- 业务策略、增长方案
- 任何需要"从多个方案中选最优"的决策

### 前置准备（必做）

**Step 0.1：探索项目上下文**
- 检查当前项目的文件、文档、最近 commit
- 了解现有架构、技术栈、业务状态
- 识别已有的约束和依赖

**Step 0.1.5：认知对齐前置检查**
- 如果任务涉及多子系统链路、阶段性规划或用户输入包含多个阶段号/模块名，先调用 `cognitive-alignment` Skill
- 如果 cognitive-alignment 已在本轮对话中完成，直接复用其输出（意图原文、问题层级、证据时间线、关键词扩展）
- 如果任务是单一明确的决策问题（如"选 A 还是 B"），跳过此步

**Step 0.2：提出澄清问题**
- 每次一个问题，优先选择题
- 重点理解：目的、约束、成功标准、时间线
- 信息足够时停止追问，用"我先假设 X"推进

**Step 0.3：启动并行调研子代理**

在开始三轮思考之前，**必须**启动子代理调研。这是 Deep Brainstorm 与普通头脑风暴的核心区别。

**强制随机约束**（防止检索偏见）：
- 至少 1 个子代理必须搜索项目代码/配置（不是文档），重点关注枚举类型、seed pack、映射表、验证器
- 至少 1 个子代理必须搜索与当前任务无直接关系的外部领域
- 所有子代理的搜索关键词不得包含用户输入中的显性关键词
- 噪声是特性不是 bug：3 个子代理中有 1 个返回关键发现就值回成本

使用 `Agent` 工具启动 **2-3 个并行子代理**，分别负责：

**子代理 1：GitHub 竞品/开源方案调研**
```
你是一个技术调研专家。请搜索与以下方案相关的开源项目和竞品。

## 当前方案关键词
<从用户需求中提取核心技术词>

## 搜索任务
1. 用 `gh search repos` 搜索相关开源项目（关注 stars > 100）
2. 用 `gh search code` 搜索关键实现模式
3. 用 WebSearch 搜索"<关键词> open source alternative"和"<关键词> github"

## 输出
- 项目名、stars、核心实现方式、是否可复用
- 与我们方案的差异点
- 值得借鉴的设计决策
```

**子代理 2：行业/商业调研**
```
你是一个行业分析师。请调研与以下业务方向相关的市场信息。

## 当前业务方向
<从用户需求中提取业务关键词>

## 搜索任务
1. 用 WebSearch 搜索"<关键词> market size 2025 2026"
2. 用 WebSearch 搜索"<关键词> business model"
3. 用 WebSearch 搜索"<关键词> competitors landscape"
4. 用 WebFetch 读取关键行业报告或竞品页面

## 输出
- 市场规模和趋势
- 主要玩家和他们的定位
- 已知的商业模式和定价策略
- 成功/失败案例及原因
```

**子代理 3（可选）：技术文档/API 调研**
```
你是一个技术文档专家。请调研以下技术方案的最新文档和最佳实践。

## 技术方案
<涉及的技术栈/API/框架>

## 搜索任务
1. 用 Context7 查询相关库/框架的最新文档
2. 用 WebSearch 搜索"<技术> best practices 2025 2026"
3. 用 WebSearch 搜索"<技术> architecture patterns"

## 输出
- 最新 API 变化和版本要求
- 推荐的架构模式
- 已知的坑点和限制
- 性能/成本对比数据
```

**Step 0.4：假设合成 + 定向反证（3+1 Scout 模式）**

当 3 个 scout 全部返回后，**禁止直接进入三轮思考**。必须执行以下步骤：

**0.4.1 合成假设**

基于 3 个 scout 的发现，用一句话提出一个方向性假设。格式：

```
基于 [scout-A 的发现] + [scout-B 的发现] + [scout-C 的发现]，
我认为 [假设内容]
```

假设必须是三个随机发现的**交叉点**，不是任何一个 scout 直接给出的。

**0.4.2 派第 4 个 scout（定向反证）**

启动第 4 个子代理，任务是：

> "找到这个假设成立所依赖的、但从未被明确质疑过的隐含前提。
>  这些前提可能是架构决策、行业惯例、团队共识、或技术假设。
>  找到它们，然后问：如果这个前提不成立，假设会怎样？"

**禁止**第 4 个 scout 去"找支持假设的证据"（确认偏误）。
也**禁止**第 4 个 scout 去"找失败场景"（太宽泛）。
准确定位是：**找假设最依赖的隐含前提**。

**0.4.3 证据分层**

第 4 个 scout 返回后，对每条发现显式标注其性质：

| 分类 | 含义 | 处理方式 |
|------|------|---------|
| 事实 | 不可改变的技术/物理约束 | 作为硬约束 |
| 现状 | 当前实现状态，可以改变 | 标注为可变 |
| 惯例 | 行业/团队惯例，可以质疑 | 作为可质疑的选择压力 |
| 假设 | 被当作前提但从未验证 | 作为可能的突破口 |

**禁止**将所有反面证据一视同仁地当作"方案不可行的理由"。

---

**调研结果整合**：子代理完成后，将发现整合到后续三轮思考中：
- 如果有成熟方案可直接复用，在第一轮中标注
- 如果有已知坑点，在第二轮中作为选择压力
- 如果有更好架构模式，在第三轮中调整方案

**碎片池持久化**：所有子代理的调研发现和后续三轮思考中产生的碎片，写入持久化文件：
`.superpowers/brainstorm/<session-date>-<topic>-fragments.json`
碎片池在整个 skill 链路中持久可用，不随方向选择丢失。后续任何步骤（包括其他 skill）都可以从池中取用碎片。

## 1. 第一轮：变异（Variation）—— 多样化生成

> 生物学对应：种群变异 + 适应辐射。进入新生态位时，物种快速分化为多种形态。

**目标：** 不急于找到"正确答案"，而是生成最大多样性的方案种群。每一个方案都是一个变异个体，争抢不同的生态位。

### 1.1 生态位测绘
- 这个决策处于什么**生态位**？（行业、用户群、技术约束、预算、时间线）
- 该生态位的**选择压力**是什么？（用户核心需求 = 生存条件）
- **已占据生态位**（常见方案/竞品）和**空生态位**（未被开发的可能方向）
- 边界条件：哪些方案注定无法存活？（违反技术约束、超出预算、与已有系统矛盾）

### 1.2 变异生成（最少 4 个方案）
强制生成至少 4 个方向不同的方案。禁止生成"同一方向的不同程度版本"——每个方案必须占据不同的生态位。

| 方案 | 生态位 | 核心区别 |
|------|--------|----------|
| V1 | 主流方案 | 最符合行业惯例的方案 |
| V2 | 邻近方案 | 同领域但换了核心策略 |
| V3 | 空位方案 | 行业内少见的、未被开发的可能 |
| V4 | 突变方案 | 打破一个行业惯例的大胆尝试 |

每个方案用**一句话**说清楚它的核心选择（"这个方案选择了让 X 用户在 Y 场景下通过 Z 方式得到 W 结果"）。

### 1.3 创始者效应检查
- 识别原始输入中的**隐含创始假设**（用户未说但已假定的前提）
- 这些假设是在打开还是关闭方案空间？
- 如果假设过窄，标注需要向用户确认的假设

### 1.4 适应度函数定义
定义本轮的**选择标准**：
- **硬约束**（存活底线）：必须满足的条件
- **加分维度**（适应度提升）：更好的方面
- **减分维度**（适应度惩罚）：风险和代价

### 1.5 调研整合
- 将子代理调研结果中的关键发现标注到相关方案上
- 如果有成熟的开源方案可复用，标注到对应方案
- 如果有已知坑点，标注到对应方案的风险栏

**输出格式：**
```
[VARIATION]
生态位: <行业/用户群/技术约束>
选择压力: <核心需求>
已占据: <常见方案> / 空位: <未开发可能>
调研发现: <子代理调研的关键发现摘要>
方案:
  V1(主流): <一句话核心选择>
  V2(邻近): <一句话核心选择>
  V3(空位): <一句话核心选择>
  V4(突变): <一句话核心选择>
创始假设: <隐含假设列表>
适应度函数: 硬约束=<底线> / 加分=<维度> / 减分=<维度>
```

### 1.6 结构化碎片输出（可选，供 MCP 消费）

如果当前会话有 cognition session 或需要持久化碎片，在自由文本输出之后追加结构化 JSON：

```json
{
  "phase": "diverge",
  "fragments": [
    {
      "text": "<碎片内容>",
      "fragmentType": "observation | analogy | contradiction | emotion | scene | mechanism | risk | edge_case",
      "source": "agent | user | tool | memory | external",
      "tags": ["<标签>"]
    }
  ],
  "candidates": [
    {
      "id": "V1",
      "niche": "<生态位>",
      "oneLiner": "<一句话核心选择>",
      "fragmentRefs": ["<引用的碎片 text 前 20 字>"]
    }
  ]
}
```

不要为了输出 JSON 而省略自由文本。JSON 是附加产物，不替代人类可读的分析。

## 2. 第二轮：选择（Selection）—— 压力测试与灭绝

> 生物学对应：自然选择。不是所有变异都能存活。选择压力淘汰不适应的个体。

**目标：** 对第一轮的所有方案施加真实的选择压力。灭绝无法存活的方案。这不是微调，是淘汰。

**强制要求：重新阅读用户的原始需求。** 对应循环深度推理的输入注入——重新注入原始输入，防止推理漂移到与用户目标无关的方向。

### 2.1 目标重注入
- **重新阅读用户的原始请求**
- 第一轮的 4 个方案是否都还在回应原始目标？
- 如有偏离，标记为**灭绝候选**

### 2.2 因果压力测试
对每个存活的方案执行因果链验证：
- 方案的核心逻辑是否有**因果关系**？（不是"因为行业都这么做"）
- 从用户需求到方案产出的因果链是否连贯？
- 是否存在**虚假因果**（相关不等于因果、幸存者偏差、类比不当）？

**断裂的因果链 = 致命缺陷。该方案灭绝。**

### 2.2.5 证据分层

对每个方案遇到的反面证据，显式标注其性质：

- **事实**：不可改变的技术/物理约束（如"LLM 无法 import TypeScript 函数"）
- **现状**：当前实现状态，可以改变（如"computeFateGravity 目前无消费者"）
- **惯例**：行业/团队惯例，可以质疑（如"direction-validator 没有用 shared 包"）
- **假设**：被当作前提但从未验证（如"三体架构是阴，应该隐藏"）

禁止将所有反面证据一视同仁地当作"方案不可行的理由"。事实是硬约束，现状/惯例/假设是可以被重新审视的选择压力。

### 2.3 成本-收益压力测试
对每个存活的方案：
- **开发成本**：人力、时间、资金
- **机会成本**：选了这个方案，放弃了什么？
- **维护成本**：长期运营的代价
- **风险成本**：失败概率 × 失败代价

**成本远超收益 = 适应度不足。降级或灭绝。**

### 2.4 共演化检测
- 方案 ↔ 用户需求是否共演化？（方案解决了需求，但需求会随方案变化）
- 技术选型 ↔ 业务目标是否共演化？（技术约束影响业务可能性）
- 如果某个方案的要素之间是**静态的**（技术不驱动业务、业务不驱动技术迭代），降级

### 2.5 局部最优陷阱检测
- 第一轮最"安全"的方案（V1）是否只是一个局部最优？
- 是否存在需要**跨过短期成本**才能到达的更优方案？
- 检查被标记为"大胆"的 V4 是否反而是更优的远程高峰

### 2.6 落地性测试（反高概念选择压力）
对每个存活方案执行**可执行性测试**：
- 能否在**一句话**内描述这个方案的第一步具体动作？
- 实施这个方案需要超过 3 个前置条件才能启动吗？如果需要，方案太"高"了
- 方案的核心价值能否用一个**可衡量的指标**表达？（不是"提升用户体验"，而是"把操作步骤从 5 步减到 2 步"）

**无法通过落地性测试 = 高概念寄生。灭绝。**

### 2.7 灭绝与留存
- 清点经过 2.1-2.6 后存活的方案
- 对灭绝的方案记录**灭绝原因**
- 对存活的方案标注**存活优势**
- 如果全部灭绝：回到 1.2 强制生成新一轮变异

### 2.7.1 discarded_trait 特征回收（必做）

灭绝方案不是废物——它们包含在当前生态位中无法存活但在其他生态位中可能有价值的特征。

对每个灭绝方案：
- 提取 1-3 个"有价值但在当前方案中无法存活"的特征
- 标记为 `discarded_trait`，写入碎片池
- 在第三轮适应和后续 per-niche 适应轮中，存活方案可以吸收这些特征

碎片格式：
```json
{
  "text": "<特征描述>",
  "fragmentType": "discarded_trait",
  "source": "extinct_V2",
  "tags": ["<来源方案的生态位>", "<特征类型>"],
  "extinctionReason": "<为什么在原方案中无法存活>"
}
```

生物学类比：进化征用（exaptation）——鸟类的羽毛最初是为了保温，后来被征用为飞行工具。灭绝方案中的特征可能在新的生态位中获得全新用途。

**输出格式：**
```
[SELECTION]
目标偏移: <偏离的方案>
因果测试: V1=<通过/断裂:位置> / V2=... / V3=... / V4=...
成本测试: V1=<成本/收益比> / V2=... / V3=... / V4=...
```

## 3. 第三轮：适应（Adaptation）—— 具体化与收敛

> 生物学对应：留存后的适应精细化 + 扩展适应（为某目的演化的特征被重新用于另一目的）。

**目标：** 将存活的最强方案从抽象思路转化为具体的、可执行的实施计划。

### 3.1 反套路扫描
存活方案中哪些元素是**行业套路**（看起来像好方案但内核空洞）？
- 用概念堆砌代替具体设计
- 用"最佳实践"代替针对性分析
- 用通用模板代替定制方案
- 用愿景代替路径

### 3.2 扩展适应搜索
已有资源/技术/能力能否被**扩展适应**为意想不到的新用途？
- 现有系统中已有的组件，能否复用于新方案？
- 已有的用户数据/行为模式，能否支撑新方案的某个环节？
- 已有的技术栈，是否有未被充分利用的能力？
- 强制寻找至少**一个扩展适应机会**
- **discarded_trait 吸收**：检查第二轮灭绝时回收的 discarded_trait 碎片，是否有特征可以被存活方案吸收。如果某个灭绝方案的特征在存活方案的生态位中反而更适应，将其整合进来。这是扩展适应的优先来源。

### 3.3 具体化替换
对存活方案执行"人-场-动-果"的具体化：
- **人**：谁执行？谁受益？谁受影响？
- **场**：在什么场景/环境/条件下？
- **动**：第一步做什么？第二步做什么？具体到可执行
- **果**：预期产出是什么？可衡量的指标是什么？

### 3.4 收敛演化验证
被灭绝的方案中是否有与存活方案**收敛**到相同洞察的？
- 如果有，这个收敛点很可能是**核心真相**——必须保留
- 如果没有，说明存活方案可能是局部最优——重新检查 2.5

### 3.5 实施路径设计
将最终方案转化为可执行的实施路径：
- **Phase 1（最小可行验证）**：最小成本验证核心假设的动作序列
- **Phase 2（扩展验证）**：验证扩展性和可持续性
- **Phase 3（规模化）**：放大已验证的方案
- 每个 Phase 包含：具体动作、预期产出、成功标准、退出条件

### 3.6 最终综合
- 将存活方案 + 扩展适应发现 + 收敛洞察综合为一个连贯的实施计划
- 标注**最强适应点**（方案的核心竞争力）和**脆弱点**（最容易失效的部分 + 应对策略）

**输出格式：**
```
[ADAPTATION]
套路清除: <清除了哪些行业套路>
扩展适应: <已有资源的新用途>
具体化: <人/场/动/果的替换记录>
收敛验证: <多个方案是否收敛到同一洞察>
实施路径:
  Phase 1: <最小验证动作> → <预期产出> → <成功标准>
  Phase 2: <扩展验证动作> → <预期产出> → <成功标准>
  Phase 3: <规模化动作> → <预期产出> → <成功标准>
最终方案: <综合实施计划>
最强适应点: <核心竞争力>
脆弱点: <最容易失效的部分 + 应对策略>
```

### 3.7 结构化适应输出（可选，供 MCP 消费）

```json

  "finalCandidate": {
    "id": "V3",
    "concretization": {
```

## 4. 跨轮次约束

### 4.1 禁止事项
- **禁止跳过任何轮次**：三轮必须依次完成
- **禁止合并轮次**：每轮有独立的任务（变异 → 选择 → 适应），不可简化
- **禁止第二轮跳过目标重注入**：必须重新阅读原始需求
- **禁止第一轮只生成 2-3 个方案**：至少 4 个，且必须占据不同生态位
- **禁止所有方案全部存活**：第二轮必须至少灭绝 1 个
- **禁止只给理论框架不给具体方案**：每轮都必须输出具体内容
- **禁止用结构化 JSON 替代自由文本分析**：JSON 是附加产物，不是主输出。如果只输出 JSON 而省略了人类可读的分析过程，视为违规。
- **禁止跳过扩展适应搜索**：第三轮 3.2 是必做步骤
- **禁止输出高概念**：任何轮次的输出中不得出现抽象战略、行业标签、概念堆砌作为方案本身

### 4.2 轮次间传递
- 第一轮的未解决疑问必须在第二轮回应
- 第二轮的新发现必须在第三轮被处理
- 第二轮的灭绝原因和存活优势必须在第三轮体现
- 如果第二轮全部灭绝，回到第一轮 1.2 重新生成

### 4.3 深度递增原则
三轮思考的深度必须递增：
- 第一轮：广度优先，多样性最大化
- 第二轮：压力测试，淘汰不适应
- 第三轮：具体化，与生态位精确匹配

**文档结构**：

## 6. 与 brainstorming 的区别

| 维度 | brainstorming | deep-brainstorm |
|------|---------------|-----------------|
| 思考轮次 | 单轮（探索 → 方案 → 设计） | 三轮（变异 → 选择 → 适应） |
| 方案数量 | 2-3 种 | 至少 4 种，必须占据不同生态位 |
| 调研 | 1 个 MiniMax 子代理 | 2-3 个并行子代理（GitHub + 行业 + 技术文档） |
| 淘汰机制 | 无 | 强制灭绝至少 1 个方案 |
| 反高概念 | 无 | 全轮次铁律 |
| 适用场景 | 功能设计、UI/UX | 产品方向、技术架构、商业模式、业务策略 |
| 输出 | 设计文档 | 三轮完整过程 + 设计文档 |
| 后续动作 | 调用 writing-plans | 调用 writing-plans |

**选择指南**：
- 如果是"怎么做"（实现层面） → 用 `brainstorming`
- 如果是"做什么/选哪个"（决策层面） → 用 `deep-brainstorm`

## 7. 边界条件与降级策略

### 7.1 调研失败降级
| 情况 | 处理 |
|------|------|
| 子代理超时（>5min 无结果） | 跳过该维度，在第一轮中标注「信息不足」，适应度函数中降低相关权重 |
| 全部子代理失败 | 跳过调研，直接进入三轮思考，在设计文档中注明「未调研」 |
| 调研结果与方案矛盾 | 在第二轮选择中作为额外选择压力，矛盾方案降级处理 |

### 7.2 方案异常处理
| 情况 | 处理 |
|------|------|
| 只能想到 2 个方案 | 强制用「如果行业惯例完全相反会怎样」「如果预算翻倍/减半会怎样」生成第 3-4 个 |
| 4 个方案高度相似 | 检查是否被创始假设锁死，放宽假设后重新生成 |
| 第二轮全部灭绝 | 回到第一轮 1.2，用灭绝原因作为新约束重新辐射 |
| 用户中途改需求 | 重注入新需求，从第一轮重新开始 |

### 7.3 时间压力处理
| 情况 | 处理 |
|------|------|
| 用户说「快点」 | 跳过子代理调研，三轮压缩为每轮 1 段输出，不写设计文档 |
| 用户说「就选 X」 | 跳过第一轮变异，直接对 X 做第二轮压力测试 + 第三轮具体化 |

### 7.4 高概念用户输入
如果用户的需求本身就是高概念（如「做一个颠覆行业的平台」）：
- 第一步：用反高概念铁律拆解用户输入，列出具体化后的需求清单
- 第二步：确认清单后再启动三轮思考
- 不要带着高概念输入直接进入变异阶段

## 9. 完整示例：为内容创作者选择变现平台

**用户输入**：「我是一个中文技术博主，有 2 万粉丝，想开始变现。帮我分析该选什么平台。」

### Step 0：前置准备
- 项目上下文：用户是技术博主，中文内容，2 万粉丝
- 澄清问题：「你的内容形式主要是？A) 长文教程 B) 短视频 C) 图文混排 D) 都有」→ 用户选 A
- 子代理调研：
  - 子代理 1（GitHub）：搜索开源付费墙/会员系统方案
  - 子代理 2（行业）：搜索「中文技术博主变现 2026」「知识付费平台对比」

### 第一轮：变异

```
[VARIATION]
生态位: 中文技术长文 / 2万粉 / 个人博主（无团队）
选择压力: 变现效率 + 内容控制权 + 运营成本低
已占据: 知识星球、公众号付费、掘金小册 / 空位: 独立站+订阅制
调研发现: 知识星球续费率普遍<30%；掘金小册需要大量前期投入；独立站需要技术能力（用户恰好有）

方案:
  V1(主流): 知识星球 — 把粉丝导入星球，按年付费，提供独家内容+答疑
  V2(邻近): 掘金小册 — 写一本系统化技术小册，一次性付费
  V3(空位): 独立站+Stripe订阅 — 用 GitHub Pages/Hugo 搭建，Stripe 收订阅费
  V4(突变): 免费内容+付费工具 — 文章全免费，但配套的代码模板/CLI工具收费

创始假设: 假设「变现=卖内容」— 但卖工具/服务也是变现
适应度函数: 硬约束=一个人能运营 / 加分=复利效应强 / 减分=前期投入大
```

### 第二轮：选择

```
[SELECTION]
目标偏移: 无（全部回应「技术博主变现」）
因果测试:
  V1: 通过 — 粉丝→星球→付费→留存，因果链清晰
  V2: 通过 — 内容质量→购买→口碑，但前期投入高
  V3: 通过 — 技术能力→搭建→订阅→复利，但获客是瓶颈
  V4: 通过 — 文章引流→工具付费，但需要额外开发成本
成本测试:
  V1: 低（星球零成本启动）但星球抽成 5-15%
  V2: 高（写小册需要 2-4 周全职投入）
  V3: 中（建站 1-2 天，但 Stripe 接入需要调试）
  V4: 高（工具开发需要额外 1-2 周）
共演化:
  V1: 静态（星球功能固定，不随内容演化）
  V3: 动态（独立站可以随内容迭代，技术栈可以复利）
  V4: 动态（工具用户反馈驱动迭代）
局部最优: V1 是最安全的，但知识星球续费率<30% 是已知坑点
落地性:
  V1: 第一步=发一条朋友圈「我开了星球，链接在这」→ 可执行
  V2: 第一步=选定小册主题，写 3 篇样章 → 周期太长
  V3: 第一步=选模板，1 小时内上线 → 可执行
  V4: 第一步=把已有代码整理成可分发的模板 → 可执行但需要额外工作
灭绝: V2 — 原因：前期投入 2-4 周，与「一个人能运营」硬约束冲突
存活: V1(弱·安全牌) / V3(强·复利) / V4(中·需开发)
最强竞争者: V3 — 理由：因果链硬 + 与技术博主身份共演化 + 复利效应最强
新发现: V3 和 V4 可以组合——独立站展示文章，文章底部推广付费工具
```

### 第三轮：适应

```
[ADAPTATION]
套路清除: V1 的「知识星球」是行业套路——「开星球」已成条件反射，但续费率<30%说明用户留存差
扩展适应: 用户的技术能力（建站）→ 扩展适应为 V3 的核心优势（别人做不了独立站，他能）
具体化:
  人: 不是「博主」，是「一个写 Rust 教程的开发者，每周发 2 篇长文」
  场: 不是「独立站」，是「用 Hugo 搭的博客，部署在 Cloudflare Pages，域名是 rust-tips.dev」
  动: 不是「开始变现」，是「在每篇文章末尾加一个卡片：完整代码模板 $9.9/月」
  果: 不是「赚钱」，是「第一个月 50 个付费用户 × $9.9 = $495，够覆盖域名+咖啡钱」
收敛验证: V3 和 V4 收敛到「技术能力是变现杠杆」——这个洞察定为核心策略
实施路径:
  Phase 1（第1周）: 用 Hugo + Cloudflare Pages 搭建独立站，部署已有文章
    → 成功标准: 站点上线，Google Analytics 接入
    → 退出条件: 如果 1 周内搞不定部署，退回 V1
  Phase 2（第2-3周）: 在文章末尾加付费卡片（Stripe Payment Link），提供 3 个代码模板
    → 成功标准: 第一个付费用户
    → 退出条件: 如果 3 周内 0 付费，分析是流量问题还是定价问题
  Phase 3（第4周+）: 根据付费用户反馈迭代工具，考虑加订阅制
    → 成功标准: 月收入 > $200
    → 退出条件: 如果连续 2 个月 <$50，考虑切换到知识星球

最终方案: 独立站(Hugo+Cloudflare) + Stripe Payment Link + 代码模板付费
最强适应点: 技术能力=建站壁垒（非技术博主做不到）
脆弱点: 获客依赖 SEO 和口碑，没有平台流量扶持 → 应对：前 3 个月在掘金/知乎引流到独立站
```

**设计文档输出**：写入 `docs/superpowers/specs/2026-04-26-content-monetization-design.md`

## 10. 第二个示例：为 SaaS 产品选择实时通信架构（技术决策类）

**用户输入**：「我们的协作编辑器需要加实时同步功能，团队 3 个后端工程师，在纠结用 WebSocket、SSE 还是 CRDT。」

### Step 0：前置准备
- 项目上下文：协作编辑器（类 Notion），已有 REST API，3 人后端团队
- 澄清问题：「实时同步的粒度是？A) 字符级（像 Google Docs）B) 块级（像 Notion）C) 仅光标/在线状态」→ 用户选 B
- 子代理调研：
  - 子代理 1（GitHub）：搜索 `crdt library`、`operational transform typescript`，发现 Yjs (35k stars)、Automerge (12k stars)
  - 子代理 2（行业）：搜索「协作编辑器架构 2026」「block-level sync vs crdt」
  - 子代理 3（技术文档）：查询 Yjs 文档、WebSocket vs SSE 性能对比

### 第一轮：变异

```
[VARIATION]
生态位: 块级实时同步 / 3 人后端团队 / 已有 REST API / TypeScript 技术栈
选择压力: 工程复杂度可控 + 冲突解决可靠 + 延迟 <500ms
已占据: WebSocket+OT（Google Docs 模式）、CRDT（Figma 模式）/ 空位: SSE+块锁
调研发现: Yjs 生态最成熟(35k stars)，但 CRDT 学习曲线陡；WebSocket 需要维护连接状态；SSE 单向但简单

方案:
  V1(主流): WebSocket + Operational Transform — 服务端维护操作序列，客户端变换后合并
  V2(邻近): WebSocket + Yjs(CRDT) — 用 Yjs 做冲突解决，WebSocket 做传输
  V3(空位): SSE + 块级锁 — 用 SSE 推送变更，块级别加乐观锁，冲突时提示用户
  V4(突变): 纯 CRDT 无服务端状态 — 用 Automerge，客户端直接 P2P 同步，服务端只存文档快照

创始假设: 假设「实时同步=需要实时传输协议」— 但块级同步不一定需要双向实时
适应度函数: 硬约束=3 人团队能维护 / 加分=与现有 REST API 共存 / 减分=引入全新基础设施
```

### 第二轮：选择

```
[SELECTION]
目标偏移: 无
因果测试:
  V1: 通过 — OT 有成熟理论，但实现复杂度高（Google 有 20 人团队做这个）
  V2: 通过 — Yjs 处理冲突，WebSocket 做传输，因果链清晰
  V3: 通过 — 块锁避免冲突而非解决冲突，架构最简单
  V4: 断裂 — P2P 需要 NAT 穿透/信令服务器，与「3 人团队」硬约束冲突
成本测试:
  V1: 高 — OT 实现需要 2-3 个月，3 人团队几乎全部投入
  V2: 中 — Yjs 集成需要 2-4 周，但学习曲线陡
  V3: 低 — SSE+块锁 1 周可上线，但体验不如 OT/CRDT 平滑
  V4: 高 — P2P 基础设施搭建 + 调试 > 2 个月
共演化:
  V2: 动态 — Yjs 的 Awareness 协议可以顺便解决在线状态/光标
  V3: 动态 — 块锁机制可以渐进升级到 CRDT（如果未来需要字符级）
  V1: 静态 — OT 与现有 REST API 是两套独立系统
局部最优: V3 最安全但体验上限低——块锁在多人同时编辑同一块时会有明显冲突提示
落地性:
  V1: 第一步=实现 OT 变换函数 → 需要 2 周理论学习，不可执行
  V2: 第一步=npm install yjs y-websocket → 可执行
  V3: 第一步=给 blocks 表加 version 字段 + SSE endpoint → 可执行
  V4: 第一步=评估 NAT 穿透方案 → 阻塞项太多
灭绝: V4 — 原因：P2P 基础设施与 3 人团队硬约束冲突
      V1 — 原因：OT 实现成本是 V2 的 3 倍，且 Yjs 已封装了 OT
存活: V2(强·最佳平衡) / V3(中·最简方案)
最强竞争者: V2 — 理由：Yjs 解决了 V1 的复杂度问题，同时保留了 V3 缺少的平滑体验
新发现: V3 可以作为 V2 的降级方案——先用 V3 上线，再渐进迁移到 V2
```

### 第三轮：适应

```
[ADAPTATION]
套路清除: 「用 WebSocket」是行业条件反射——但块级同步用 SSE+长轮询已经够用，WebSocket 不是必须的
扩展适应: 现有 REST API 的文档版本号字段 → 扩展适应为块锁的 version 字段，零成本复用
具体化:
  人: 不是「后端工程师」，是「3 个写 TypeScript 的全栈，没做过实时系统」
  场: 不是「协作编辑器」，是「一个类似 Notion 的产品，200 个日活，同时编辑同一文档的峰值 5 人」
  动: 不是「加实时同步」，是「用户 A 编辑第 3 块时，用户 B 的屏幕在 300ms 内更新」
  果: 不是「支持实时协作」，是「5 人同时编辑时冲突率 <2%，用户无感知」
收敛验证: V2 和 V3 收敛到「块级同步的核心是冲突避免而非冲突解决」——这个洞察定为架构原则
实施路径:
  Phase 1（第1周）: 集成 Yjs + y-websocket，实现块级文档同步
    → 成功标准: 2 个浏览器窗口实时同步同一文档
    → 退出条件: 如果 Yjs 与现有数据模型不兼容，降级到 V3
  Phase 2（第2周）: 加 Awareness（在线状态、光标位置）、处理断线重连
    → 成功标准: 断网 30s 后自动重连，无数据丢失
    → 退出条件: 如果重连丢数据率 >1%，回退到轮询方案
  Phase 3（第3-4周）: 压力测试（模拟 20 人同时编辑）、优化性能
    → 成功标准: p99 延迟 <500ms，CPU 占用 <30%
    → 退出条件: 如果 Yjs 内存占用过大，考虑切换到 y-websocket 的分片模式

最终方案: Yjs + y-websocket（块级同步）+ REST API（持久化）
最强适应点: Yjs 的 Awareness 协议同时解决了在线状态，零额外成本
脆弱点: Yjs 的 Y.Doc 内存占用随文档大小线性增长 → 应对：大文档按块分片加载
```

**设计文档输出**：写入 `docs/superpowers/specs/2026-04-26-realtime-sync-architecture.md`

## 11. 用户中途改需求处理流程

如果用户在三轮思考进行中改变需求：

### 触发信号
- 用户说「我改主意了」「其实我要的是...」「换个方向」
- 用户推翻了之前确认的约束或假设

### 处理步骤
1. **暂停当前轮次**，不继续输出
2. **重注入新需求**：用一句话复述用户的新需求，确认理解正确
3. **评估影响范围**：
   - 如果新需求只影响适应度函数（选择标准变了）→ 从第二轮重新开始
   - 如果新需求改变了生态位（行业/用户群/约束变了）→ 从第一轮重新开始
   - 如果新需求只是微调（某个方案的细节变了）→ 在当前轮次内调整
4. **保留可复用部分**：已经灭绝的方案和灭绝原因仍然有效，不需要重新生成
5. **告知用户**：「需求变更影响了 [X]，我从第 [N] 轮重新开始」

### 示例
```
用户：「其实我不要独立站，太麻烦了，还是用平台吧」
→ 影响：V3 被排除，适应度函数中「技术门槛」权重降低
→ 处理：从第二轮重新选择（V1 知识星球 vs V4 付费工具）
→ 保留：V2 的灭绝原因（前期投入大）仍然有效
```
