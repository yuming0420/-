# 调研与设计文档产出（Research-Spec）

产出调研/设计文档的技能。终态是 spec 文档落地——**不强制跳转到执行计划**。

与 `writing-plans` 的区别：
- `research-spec` → 产出 **spec 文档**（问题建模、方案对比、假设验证），存 `docs/superpowers/specs/`
- `writing-plans` → 产出 **执行计划**（Task/TDD/验证命令），存 `docs/superpowers/plans/`

用户在 spec 产出后可以选择：继续精化 spec、调用 writing-plans 生成执行计划、或到此为止。

---

## 流程

### 阶段 1：问题建模

用**系统诊断语言**描述问题，不用解决方案语言。

- 列出所有独立的执行路径/门控点，标注它们**共享的缺口**
- 引用真实文件路径和函数名（`read_file`/`grep`/`glob` 自行调研）
- 回答：问题为什么存在？影响面多大？谁被影响？

**反模式**：
- ❌ "我们需要一个 X 功能"
- ✅ "两个独立门（A + B）都缺少 Y 能力。今天用户做 Z 时，两个门都硬拒绝。"

### 阶段 2：调研（自行读代码 + 可派 scout）

#### 2.1 自行调研（必做）
- `read_file` 所有相关源文件（包括工具函数和类型定义）
- `grep` 调用者和依赖关系
- 读测试文件理解行为边界
- `git log --oneline -10 -- <file>` 理解修改历史

#### 2.2 派 scout 子代理（可选，问题跨域时推荐）
用 `delegate_task(kind="doc_research", profile="doc_scout")` 并行调研：
- Scout A：代码架构审计（grep 调用链、类型流）
- Scout B：外部参考/竞品调研
- Scout C：数据/测试缺口分析

#### 2.3 调研完整性自检
- [ ] 已读所有相关文件（包括工具函数和类型定义）
- [ ] 已理解每个拟修改函数的存在原因
- [ ] 已 grep 所有调用者（对删除/修改操作）
- [ ] 已检查是否有现有实现已经解决问题

### 阶段 3：方案生成与选择

生成 2-4 个方向不同的方案（不是同一方向的程度变体），每个写清楚：
- 核心选择（一句话）
- 优势和代价
- 风险点

**选择标准**：
- 硬约束（存活底线）
- 因果链是否连贯
- 落地性（第一步具体动作是什么）

**淘汰**弱方案，标注淘汰原因。记录灭绝方案中可回收的特征。

### 阶段 4：设计具体化

对存活方案执行具体化：
- **边界标定**：显式列出"会碰哪些文件"和**不改什么**
- **注入点选择**：每个改动的 当前行为 → 改后行为 → 为什么安全
- **数据流**：画出状态的生产者、消费者、存储（Mermaid 可选）
- **先例引用**：代码库中已有的类似模式

### 阶段 5：文档输出

**路径**：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

**文档结构**：
```markdown
# <主题> — 设计文档

> 日期：YYYY-MM-DD
> 状态：设计已定稿 / 待审查 / 已执行

## 背景与问题
<问题建模 + 影响面>

## 调研发现
<关键发现摘要 + scout 结果>

## 方案对比
| 方案 | 核心选择 | 优势 | 代价 | 结论 |
|------|----------|------|------|------|
| V1 | ... | ... | ... | 存活/淘汰 |

## 最终方案
<具体设计 + 边界标定 + 注入点>

## 先例引用
<代码库中已有的类似模式>

## 风险与应对
<脆弱点 + 应对策略>

## 下一步（可选）
<如果需要执行计划，调用 writing-plans 技能>
<如果只是记录结论，到此为止>
```

### 阶段 6：自检与交付

**规格自检**：
1. 占位符扫描：有没有 TODO/TBD？
2. 内部一致性：各部分之间有矛盾吗？
3. 范围检查：是否聚焦？
4. 文件路径：全为确切路径，无模糊引用？

**交付**：
> "设计文档已保存到 `<path>`。"

**不做**：不自动跳转到 writing-plans。用户需要执行计划时会自己说。

## 与其他技能的关系

| 场景 | 用什么 |
|------|--------|
| 需要调研/探索/出 spec | **research-spec**（本技能） |
| 有 spec 了，要出执行计划 | `writing-plans` |
| 有计划了，要执行 | `executing-plans` |
| 决策层面多方案深度竞争 | `deep-brainstorm`（三轮演化） |
| 任务大需要并行委派 | `subagent-driven-development` |

---

## 关键原则

- **调研深度决定方案质量**：读得够深，方案自然不会错
- **不强跳执行**：spec 是独立产出物，不是执行计划的前奏
- **自行调研优先**：计划/调研阶段不派 worker 子代理写代码
- **具体化胜过抽象**：每个方案必须能用"谁+场景+动作+结果"表达
